# 주요 결정 기록

중요한 제품·기술 결정을 이유와 함께 남기는 문서입니다. 새 결정은 기존 번호를 바꾸지 않고 아래에 추가합니다.

상태는 `제안`, `채택`, `대체`, `폐기` 중 하나를 사용합니다.

## D-001 — 모바일 세로형 게임을 제품 기준으로 사용

- 상태: 채택
- 결정: 기준 화면을 `390×844px`, 최소 너비를 360px로 하고 PC에서도 최대 430px 앱 프레임을 가운데 표시합니다.
- 이유: 모바일에서 동일한 조작과 정보 밀도를 유지하고 PC는 테스트 가능한 보조 환경으로 사용하기 위해서입니다.
- 결과: 넓은 데스크톱 대시보드 레이아웃은 만들지 않습니다.

## D-002 — 회사 설립 직후 원룸 사무실에서 시작

- 상태: 채택
- 결정: 첫 게임 화면은 주식 목록이 아니라 원룸이며 CRT 컴퓨터를 눌러 시장에 진입합니다.
- 이유: 숫자 중심 시뮬레이션에 장소성과 성장 체감을 부여하기 위해서입니다.
- 결과: 전화와 서류함을 포함한 방의 사물이 주요 기능 진입점입니다.

## D-003 — 진행상태를 브라우저 로컬에 자동 저장

- 상태: 채택
- 결정: 회사명과 진행상태를 `localStorage` 키 `simul-millennium-capital-v1`에 저장합니다.
- 이유: 계정 없이 바로 시작하고 새로고침 후에도 진행을 유지하기 위해서입니다.
- 결과: 상태 스키마 변경에는 반드시 마이그레이션이 필요합니다.

## D-004 — 실제 일별 수정주가 흐름을 사용

- 상태: 채택
- 결정: 종목별 기준일을 100으로 정규화한 adjusted close를 사용하고 휴장일에는 직전 종가를 유지합니다.
- 이유: 역사적 수익률 흐름을 재현하면서 미래 가격 누출을 막기 위해서입니다.
- 결과: 이 값은 당시 실제 호가가 아니며 게임용 수치와 구분해 표시합니다.

## D-005 — 기존 웹 기술 구성을 유지

- 상태: 대체
- 결정: vinext, React, TypeScript 구현은 Flutter 이관이 끝날 때까지 기능 기준과 비교용으로 보존합니다.
- 이유: 기존 앱에 시장, 거래, 인수와 역사 데이터 로직이 이미 구현되어 있기 때문입니다.
- 결과: 새 기능의 주 구현은 D-008에 따라 Flutter에서 진행합니다.

## D-006 — 10살 소년을 인물 중심 주인공으로 도입

- 상태: 채택
- 결정: 2000년의 컴퓨터를 독학한 10살 소년을 플레이어의 분신으로 도입합니다.
- 이유: 투자 숫자만으로는 부족한 감정적 연결과 성장 서사를 만들기 위해서입니다.
- 결과: Flutter 온보딩과 사무실 홈에 소년 주인공을 배치했습니다.
- 미결 사항: 회사 설립의 법적·서사적 개연성과 해킹 설정의 결과를 다루는 방식

## D-007 — 2D 캐릭터 원본은 엔진 중립적으로 관리

- 상태: 채택
- 결정: 캐릭터 원본은 투명 PNG로 보관하고 런타임에서는 축소 스프라이트와 아틀라스를 사용합니다.
- 이유: 같은 원본을 Flutter와 향후 다른 엔진에서 재사용하고 큰 텍스처로 인한 성능 낭비를 막기 위해서입니다.
- 결과: 첫 주인공 PNG를 `flutter_app/assets/images/hero-boy.png`에 포함했습니다.

## D-008 — 새 주 구현을 Flutter로 전환

- 상태: 채택
- 결정: 밝은 2D UI와 모바일 앱 확장을 위해 새 구현은 Flutter와 Dart로 진행합니다.
- 이유: 화면 컴포넌트, 애니메이션과 기기 로컬 상태를 하나의 코드베이스에서 Android와 Web에 제공하기 위해서입니다.
- 대안: 기존 React 구현을 계속 확장하거나 Unity로 전환하는 방안을 검토했습니다.
- 결과: `flutter_app/`을 추가했으며 기존 React 앱은 기능 이관과 저장 데이터 마이그레이션이 끝날 때까지 삭제하지 않습니다.

## D-009 — 실제 회사명을 대체역사 콘텐츠에 사용

- 상태: 채택
- 결정: 개인용 게임 콘텐츠에서 Apple 등 실제 회사명을 사용합니다.
- 이유: 플레이어가 실제 역사와 자신의 선택 차이를 더 직접적으로 느끼게 하기 위해서입니다.
- 결과: 실제 회사명과 실제 데이터 사용을 구분합니다. 공개 근거가 없는 내부 대화, 비용, 점수, 코드명과 결과는 게임용 가상 시나리오라고 명시합니다.

## D-010 — 분기 이후 결과를 저장 가능한 결정론적 시뮬레이션으로 처리

- 상태: 채택
- 결정: 지배권 발생일 가격을 역사 기준선에 연결하고 이후 결과는 `simulationSeed`, 누적 상태와 선택으로 계산합니다.
- 이유: 실제 미래 가격 누출을 막고 같은 저장·선택의 재현성을 보장하기 위해서입니다.
- 결과: 현재 상태 스키마에 분기일, 분기 원인, 시뮬레이션 가격, 지연 사건, 결정과 원장을 저장합니다.

## D-011 — 초반을 보호자 동의 가족 투자연구소로 시작

- 상태: 채택
- 결정: 10살 주인공은 어머니 명의 교육용 계좌에서 기업 조사와 투자 제안서를 담당하고, 부모가 계좌·비밀번호·주문을 관리합니다. 초기 조직은 법인이 아니라 가족 투자연구소입니다.
- 이유: 미성년자의 법적·서사적 개연성을 확보하면서 부모 승인을 튜토리얼과 권한 성장 시스템으로 활용하기 위해서입니다.
- 결과: 새천년 콜드 오픈, 주인공 동기, 가족 규칙, A4 간판과 가족 신뢰를 저장 스키마 v5에 구현했습니다. 실제 회사 경영권과 M&A는 성장 뒤 해금합니다.

## D-012 — 가족 도움에서 정식 직원 조직으로 단계적 성장

- 상태: 채택
- 결정: 2000~2002년은 가족 조언자 배치와 피로도 시스템으로 운영하고, 2003년 첫 아르바이트 조사원부터 정식 채용을 해금합니다. 직원은 직군·등급·잠재력·업무량·사기·충성도·윤리와 핵심 능력을 가진 개별 인물로 저장합니다.
- 이유: 10살 주인공의 법적·서사적 개연성을 지키면서 작은방의 가족 도움을 이후 회사 조직 성장과 끊김 없이 연결하기 위해서입니다.
- 결과: 저장 스키마 v6에 `OrganizationState`를 추가하고, 가족별 도움·피로·기록·문화 태그와 미래 직원 모델을 구현했습니다. 배치 UI는 선택 인물의 상체 초상과 하단 선택 카드를 기준으로 사용합니다.
## D-013 — 즉시 증여 대신 0원에서 종잣돈을 직접 마련

- 상태: 채택
- 결정: 신규 플레이는 현금 0원과 빈 저금장부로 시작합니다. 만 10세에는 집안일·보호자 동행 문방구 정리·가족 벼룩장터 미니게임으로 돈을 벌고, 첫 조사예산 1만원 이후에도 계속 일할지는 플레이어가 선택합니다.
- 이유: 작은 돈의 가치와 투자 원금의 무게를 직접 체감시키고, 서사 설명이 아니라 실제 플레이로 시작 자본을 만들기 위해서입니다.
- 근거: 2000년 1월 1일~8월 31일 법정 최저임금 시간급 1,600원을 문방구 30분 수당 800원의 기준선으로 사용합니다. 만 10세는 정식 고용이 아니라 보호자 관리 활동으로 제한합니다.
- 결과: 프롤로그의 100만원 봉투를 빈 저금장부로 교체하고, 설거지 순서·문방구 분류·벼룩장터 거스름돈 3종 미니게임, 점수별 보상, 하루 3회 제한과 현금 장부 기록을 구현했습니다. 기존 저장 데이터의 현금은 보존합니다.
